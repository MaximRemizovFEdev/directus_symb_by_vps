# Symbolika Directus Clean Install

## Контекст проекта для дальнейшей разработки

Проект — локальная учетная система «Символика» на Directus 11 и PostgreSQL. Рабочая папка для разработки: `symbolika_directus_clean_install/`. Локальный адрес админки: `http://localhost:8057/admin`. Основные контейнеры: `symbolika-directus` и `symbolika-db`.

Тестовые учетные записи для проверки прав и интерфейса: администратор `dimon96af@yandex.ru`, менеджер `test@symbcorp.ru` / `test123456`. После изменений прав, layout или JS/CSS-инъекций нужно проверять результат под нужной ролью.

В Git хранить код, конфиги, setup-скрипты, расширения и документацию: `docker-compose.yml`, `extensions/`, `setup/`, `README.md`, `INSTALL_CLEAN.md`, `.env.example`. Не коммитить живую базу `database/`, логи, кеши, временные архивы и загрузки без отдельного решения. Ничего не удалять из рабочей папки без явного разрешения.

Ключевые файлы:

- `setup/create-work-views.sql` — основной SQL-слой витрин, Directus metadata, ролей, прав, layout и служебных представлений.
- `setup/import-symbolika.mjs` — чистый импорт базовой структуры и данных.
- `setup/patch-directus-admin-locale.sh` — патчит админку Directus, добавляет `lang="ru"`, `translate="no"`, `notranslate` и подключает кастомные CSS/JS.
- `setup/symbolika-admin-ui.css` и `setup/symbolika-admin-ui.js` — темная тема, визуальные правки Directus, русские статусы, пины, layout, readonly-стили, счетчики позиций и автосохранение.
- `extensions/symbolika-calculations` — бизнес-логика пересчета сумм, оплат, статусов и служебных полей.
- `extensions/symbolika-autosave-select`, `symbolika-order-link`, `symbolika-office-issue-link`, `symbolika-live-calc-interface` — интерфейсные расширения для селектов, ссылок на заказ, переходов между витринами и живых расчетов.

Основная логика ролей и витрин:

- Менеджер в обычных коллекциях видит свои заказы, свои позиции, свои платежи и своих клиентов. Компании можно читать шире, потому что сверка по компании общая.
- Витрины `Выдача в офисе` и `Заказы в офисе` доступны менеджерам, чтобы любой менеджер мог выдать заказ в офисе.
- В `Выдача в офисе` менеджеру нужны: менеджер, клиент, номер заказа, позиции заказа, сумма к оплате, статус офиса, добавление оплаты и тип оплаты. Лишние управленческие поля там не показывать.
- `Заказы в офисе` показывает позиции, которые физически в офисе. После выдачи позиция и заказ должны уходить из активных офисных витрин.
- `Архив выдачи в офисе` хранит выданные офисные заказы.
- `Мои заказы` разделены на `Заказы в работе`, `Завершенные заказы`, `Неоплаченные заказы`; есть отдельная сводная группа `Заказы` со сроками: сегодня, эта неделя, следующая неделя, этот месяц, следующий месяц.
- Производственные витрины: `Производство`, `Шелкография`, универсальная логика `Работы контрагента`. У контрагента есть признак собственного представления; в будущем это нужно использовать для доступа внешних подрядчиков.

Логика офисных статусов:

- Если все позиции заказа в офисе, заказ получает офисный статус `В офисе`.
- Если хотя бы одна позиция не в офисе, заказ получает `Не в офисе`.
- Если заказ выдан, все его позиции становятся выданными.
- Если все позиции выданы, заказ становится выданным.
- Если выданный/доставленный заказ частично возвращают в статус `В офисе`, статус заказа возвращается в рабочий `Готов`, чтобы менеджер видел, что действие еще требуется.

Логика оплат:

- Оплаты и админские финансы — разные блоки. Менеджеру нужен блок `Оплаты`, а управленческие финансовые поля должны быть доступны только администратору или управляющему.
- При добавлении оплаты через `Выдача в офисе` платеж должен автоматически создаваться для текущего заказа и распределяться на этот заказ.
- Поля `Оплачено`, `Остаток к оплате`, `К оплате при выдаче` должны считаться автоматически триггерами/расширениями, а не вручную.

UI-принципы:

- Система используется в темной теме, поэтому все кастомные стили должны проектироваться сразу под dark UI.
- Readonly-поля визуально серые. Редактируемые поля обычного цвета. Обязательные поля выделяются оранжевой обводкой.
- Статусы должны отображаться на русском и по возможности цветными пинами. Не показывать системные значения вроде `in_office`, `ready`, `not_in_office`.
- Карточки должны быть сгруппированы по смыслу: сводка, клиент и сроки, позиции, статусы, доставка, оплаты. Для широких экранов использовать 4 колонки, но не превращать карточку в хаотичную сетку.
- Витрины должны давать быстрый переход в исходный заказ только там, где это действительно нужно и где у роли есть права. В `Заказы в офисе` переход должен вести в `Выдача в офисе`, а не в обычный заказ.
- Для Chrome/Yandex важно не допускать автоперевод страницы: Directus должен восприниматься как русский интерфейс, иначе браузер ломает подписи полей.

Автосохранение:

- Нативное сохранение Directus через `Ctrl+S` работает, но имитация клика по кнопке сохранения может уводить пользователя из карточки.
- Текущий подход: кастомный JS сохраняет измененное поле напрямую через `PATCH /items/{collection}/{id}` и не нажимает кнопку Directus. Это оставляет пользователя в карточке.
- После прямого PATCH Directus может продолжать считать форму несохраненной, потому что его внутреннее состояние формы не знает о нашем API-сохранении. Поэтому `symbolika-admin-ui.js` автоматически закрывает модальное окно `Несохранённые изменения` только в течение короткого окна после успешного автосохранения, нажимая `Отменить изменения`. В этот момент значение уже записано в БД, а кнопка лишь сбрасывает устаревшее локальное dirty-состояние формы.
- Русские строки в `symbolika-admin-ui.js` лучше хранить через Unicode escape (`\u041f...`), чтобы при инъекции в Directus не появлялись кракозябры.
- После изменения `symbolika-admin-ui.css/js` нужно выполнить `docker cp`, `patch-directus-admin-locale.sh`, `docker restart symbolika-directus`, затем в браузере сделать жесткое обновление `Ctrl+F5`.

Быстрая проверка после UI-патчей:

```powershell
docker cp symbolika_directus_clean_install\setup\symbolika-admin-ui.js symbolika-directus:/directus/setup/symbolika-admin-ui.js
docker cp symbolika_directus_clean_install\setup\symbolika-admin-ui.css symbolika-directus:/directus/setup/symbolika-admin-ui.css
docker exec symbolika-directus sh /directus/setup/patch-directus-admin-locale.sh
docker restart symbolika-directus
```

## Рабочие коллекции-представления

Для отдельных ролей есть SQL-скрипт:

```bash
docker exec -i symbolika-db psql -U directus -d directus < setup/create-work-views.sql
docker restart symbolika-directus
```

Он создает коллекции:

- `Выдача в офисе` — заказы со способом отгрузки `office_pickup`.
- `Заказы в офисе` — позиции заказов, у которых `office_status = in_office`.
- `Производство` — позиции, где контрагент 1 или 2 содержит `производств`.
- `Шелкография` — позиции, где контрагент 1 или 2 содержит `шелкограф`.

Коллекции синхронизируются с основными заказами и позициями. Офис-менеджер в `Выдача в офисе` видит только офисную витрину заказа, может менять `office_status` и добавлять оплату через поля `add_payment`, `payment_type`, `payment_comment`. Вложенные позиции заказа отображаются таблицей с колонками `product_name`, `quantity`, `office_status`; статус позиции можно менять прямо из этой таблицы.

Офисный статус позиции меняется в коллекции `Заказы в офисе` или в служебной табличной витрине позиций. Менеджерские политики могут менять только `office_status` позиции своих заказов. Производство может менять только `production_status`.

В карточке позиции коллекции `Заказы в офисе` есть кнопка `Перейти в заказ`: она открывает соответствующий заказ в коллекции `Выдача в офисе`.

Коллекции `Выдача в офисе` и `Заказы в офисе` доступны всем менеджерам, чтобы любой менеджер мог выдать заказ в офисе. В обычных коллекциях `Заказы` и `Позиции заказа` менеджер видит только свои записи. В левом меню менеджера остаются рабочие разделы: свои заказы, свои позиции заказа, платежи, распределение платежей, связи клиентов, клиенты, компании, выдача в офисе и заказы в офисе; служебные справочники скрыты из меню, но остаются доступными для выпадающих списков.

Менеджер видит и редактирует только своих клиентов. Менеджер может создать нового клиента и новую компанию; текущий сотрудник-менеджер проставляется автоматически hook-расширением, даже если в запросе передали другого менеджера. Справочник компаний общий для чтения, потому что сверка по компании ведется общей. Редактирование компаний и постоянные связи клиент-компания выполняет управляющий или администратор; менеджер выбирает существующую компанию в заказе, но не меняет общую структуру связей.

Если все позиции заказа находятся в офисе или выданы, статус офиса у заказа пересчитывается автоматически. Если статус офиса меняют у заказа, такой же статус проставляется всем его позициям.

Для не-админских ролей карточка сотрудника ограничена полями `full_name`, `position`, `phone`; полную информацию о сотрудниках редактирует администратор или управляющий.

Админка Directus при старте патчится скриптом `setup/patch-directus-admin-locale.sh`: HTML получает `lang="ru"`, `translate="no"` и `notranslate`, чтобы Chrome/Yandex Browser не включали автоперевод и не ломали русские поля интерфейса.
